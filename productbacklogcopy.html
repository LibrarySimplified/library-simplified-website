Servers

Library Simplified application servers

The Library Simplified solution is a set of backend middleware servers and cloud services that support the seamless aggregation and publication of library collections to software client applications, including the SimplyE app. The application architecture adheres to some basic principles: interoperability, scalability, open source standards and technologies. As such the application uses a RESTful architectural style and utilizes the Open Publication Distribution System (OPDS catalog) protocol. OPDS is an XML expression of a library collection based on the Atom protocol. OPDS 1.2 was the beginning of the specifications entry into library lending, authentication, discovery, presentation, and distribution. The protocol also serves as a basis for Open Distribution Libraries (ODL) as an alternative interface for middleware integration into alternate commercial distribution models widely used in the global eBook market as opposed to proprietary APIs and FTP exchange. This open protocol is also the basis of communication between Library Simplified servers.

Circulation Manager

The purpose of the circulation manager is to aggregate, unify, normalize, present and transact lending of the eBook collection of a library regardless of the content source or host. The circulation manager should support licensed content and Open Access / Public Domain content. The objective is to facilitate efficiency, interoperability, accessibility of catalog. As a principle, the circulation manager should work with other client applications through utilization of industry standards and best practices.

Summary

The circulation manager must be able to keep track of the respective library's eBook collection for OD, 3M and Baker and Taylors Axis 360 collections. This integration must allow the application to understand availability of the respective titles for lending and communicate that to the end client application. The application should also use OPDS and RESTful API architecture.

Key features

  • Searchable Catalog
  • Scalable Catalog - RESTful Web application
  • Open Access Content / Public Domain
  • Licensed Content
  • Standards / Best Practices (OPDS)
  • Interoperability (3rd party ebook services and hosted content services)

Version 1.0

The circulation server is responsible for a consolidated view of every book in our catalog (whether we own that book through open-access, through a license, or both). For each book, this view includes:

  • The bibliographic information we want to display to the user, e.g. title, author, language, audience.
  • A numeric estimate of the book's quality.
  • An assignment of the book to one (and only one) lane.
  • Information about each active license for the book.
  • Information about all current activity between Simplified patrons and the book.

The circulation server is not necessarily responsible for gathering, keeping, or processing raw data (e.g. OCLC Classify records, NYT bestseller information, or Amazon reviews), or the preliminary data used to build this view (rows in the 'workrecords' table).
However, the circulation server is responsible for the search engine, which for now probably means storing that intermediate data. Search results are served as OPDS feeds. The only supported search is a combination title/author search.
The circulation server serves OPDS feeds of books written in various languages and found in various lanes. It can order these feeds by title and author. It can also generate 'recommended' feeds which include a random sampling of high-quality books for a given combination of language+lane.

Browesable Catalog - OPDS Catalogs

Groups are a feature of OPDS catalogs which allow grouping of catalogs items into logical organizations. For libraries these can be book categories such fiction, non-fiction, audience, language, genre and sub-genera, subject, best sellers or staff picks or recommendations or appeal.

Facets are a feature of OPDS catalogs which allow the user to interact with the catalog to sort or filter items in the collection. Facets can be presented to the user in the client interface or merely used by the application in general to present the collection in a useful way to drive user experience, catalog exposure, or other circulation objective.

Seachable Catalog - Materialized Views

The circulation server is responsible for the search engine. For version 1.0 the circulation manager will support the open search specification of OPDS by using SQL search at the DB level with an optimized DB scheme and materialized views.

Scalable Catalog - RESTful Web application

The Circulation Manager should be an easy to deploy web applications that can be scaled using simple load balancing technology, caching and Content Delivery Network (CDN) technologies.
The circulation Manager should communicate over Hypertext Transfer Protocol (HTTP) using HTTP verbs (GET, POST, PUT, DELETE, etc.) which web browsers use to retrieve web pages and to send data to remote servers. It should interface with external systems using resources identified by Uniform Resource Identifier (URI.)

Licensed Content

In particular, the circulation server is responsible for acting on behalf of a patron to orchestrate the process of obtaining a license file for a DRM-encrypted book, regardless of where the license server is located, who controls the license server, or which type of DRM has been applied to the book. The circulation server is responsible for telling the client where to go to get the license file and the DRM-encrypted book, but not necessarily responsible for hosting those files.

The circulation server is responsible for processing user requests for returning a book early, extending a license, or joining the hold queue for a book.

Interoperability (3rd party ebook services and hosted content services)

The circulation server is responsible for knowing about and granting access to binary files such as cover images, open-access books, DRM-encrypted books, unencrypted previews, and license files. It is not responsible for hosting any of those files; it just knows where they are.

Security and Privacy (Best Practices)

The circulation server is responsible for validating a patron's barcode and PIN with Millennium or with any successor SSO service. It is not responsible for storing a patron's barcode or PIN. This will be done of SSL. No direct connection to the ILS will be made. The database will not store credentials in an unprotected state if stored at rest, nor will it gather or store other patron information contained in the ILS or other identity management platform not needed to provide the service.

Seachable Catalog - Elastic Search

The circulation server is responsible for the search engine. For this task the circulation manager will Elastic Search. Elastic Search is a Lucene based search technology providing a distributed, multitenant-capable full-text search engine with a HTTP web interface and schema-free JSON documents. Elasticsearch is open source under the terms of the Apache License. Elasticsearch requires storing intermediate data. The circulation manager must provide the search results as OPDS feeds. Search results will match either a combination title/author/subtitle search or genre, audience, and fiction/nonfiction indicators. If there are additional search terms, a genre/audience/fiction search result will need to match the other terms in the summary, author, or subtitle.

Additional DRM support (target 2.0)
  • Readium LCP - Licensed Content Protection (LCP) is a DRM protection technology developed by the Readium Foundation and the European Digital Reading Lab and DRM Inside from Korea as a new transaction cost free solution for Reading system developers and service providers. This new technology provides robust content protection using Public Key infrastructure. The non proprietary technology allows for secure interoperability between reading system providers for properly licensed content access.
  • Sony/DADC URMS - User Rights Management System (URMS) is a commercial DRM based on the open MARLIN standard. This commercial supported technology is available to commercial and nonprofit content distributors.
Additional format support (target 2.0 and above)
  • MP3 - Audiobooks This feature will allow access to licensed content on Overdrive as well as Public Domain content on Librivox. (Target 2.0)
  • MP4 - Video This feature will allow access to licensed content on Overdrive. (Target 3.0)
  • Fixed layout (target 4.0)
    • PDF - This will allows display of non DRM content hosted on subscription services and locally hosted content. (Target 4.0)
    • Comic book archive - These files mainly consist of a series of image files, typically PNG or JPEG files, stored as a single archive file. Occasionally GIF,BMP, and TIFF files are seen. Folders may be used to group images. The filename extension indicates the archive type used: .cb7/7z, .cba/ACE, .cbr/ RAR, .cbt/TAR, .cbz/ZIP (Target 4.0+)

Recommendations -
  • Add support for “Like” based recommendations on the Book Detail view. The view should be a “lane” of similar books the user may want to read such as “Books by Same Author”, “Books in the same Series” “Staff Picks” (aka book on the staff picks list that are of the same category (Sub-genera or subject).
  • Include - Appeals based recommendations is a method derived from a machine learning experiment that uses a structured taxonomy mine the professional reviews of a work to rate it in terms of its appeal (language, story, setting or character) of a book. For now it is used to simply classify works but could be used as a basis for a recommendation engine.

Miscellaneous-
  • Page transition - Currently the app does simple page transitions. In some implementations, page transitions have been developed to provide page “slide”, or page “turn” transitions options for the user to apply to the reading experience. This needs to be refactored with Readium’s help possibly using css snap points.

  • Text to speech. For iOS, text to speech API was introduced in iOS 7 . This would allow computer voice read along of content. The call back would allow JS to be inserted on the particular word so that it could be highlighted for assisted reading. Controls for play speed, pause. UI interactions for content selection could be applied as well.
  • Bookmarks - allows the user to bookmark CFI location, name the location, bring up a list of book marks for a particular book, and navigate to that location. This includes last reading location. Consideration for synching the book mark and reading position between devices should be considered in scope.
  • Page numbering / navigation - Currently, there is not a clean way of conveying the concept of pages with reflowable content. Logically books are grouped by chapters. Pages are a physical print media concept that due to the unlimited compbination of screen size and font size permutations require dynamic page numbering. This is further confounded by the resource pacakaging and application of DRM.
